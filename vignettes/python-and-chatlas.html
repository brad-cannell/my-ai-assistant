Overview

This vignette provides information about using Python and Chatlas to interact with AI assistants. It describes the conceptual role of each step in a typical API conversation workflow, and documents less-common techniques not covered in the conversation templates. For working code for routine tasks, see the template conversation files in conversations/.

Create a Virtual Environment

Before working with Python packages, you need a virtual environment. See one of the following vignettes depending on your setup:

  • vignettes/venv-virtual-environments.qmd — standard Python venv approach
  • vignettes/uv-virtual-environments.qmd — modern uv approach, recommended for cross-machine workflows

Install Packages

See vignettes/venv-virtual-environments.qmd or vignettes/uv-virtual-environments.qmd for full installation instructions and package management guidance.

Standard Conversation Workflow

The sections below describe the conceptual role of each step in a typical API conversation. The specific packages, models, file paths, and helper functions in use at any given time can change, so the working code lives in the template conversation files in conversations/ rather than here.

Load Packages

Every conversation document begins by loading the Python packages it depends on. This typically includes packages for interacting with the API (e.g., openai, chatlas), managing credentials (e.g., keyring), handling file paths and dates, and any local helper packages. Keeping all imports at the top of the document makes dependencies explicit and makes it easy to spot when a package needs to be installed.

File Paths and Environment Variables

Declaring file paths, environment variables, and the model name near the top of the conversation document — rather than inline throughout the code — makes the document easier to reuse and adapt. When you need to point the conversation at a different file or switch models, there is a single place to make that change. This section typically sets today’s date, paths to any prompt templates or input files, and the model identifier.

Load Helper Functions

Many conversations rely on local helper functions stored in a shared Python package within the repo. Loading these functions near the top of the document, after packages and file paths, keeps the rest of the conversation clean and focused on the task rather than implementation details.

Expose API

API keys should never be hardcoded in a script or committed to Git. The standard approach in this repo is to store keys in the system keyring and retrieve them at runtime, then set them as environment variables for the duration of the session. This section handles that retrieval and confirms the key was found before proceeding. See vignettes/keyring.qmd for setup instructions.

Read the Template Prompt

When using a prompt template stored in prompts/, this step reads the template file, extracts the system prompt and user message sections separately, and injects any dynamic values (e.g., lesson name, course name) using placeholder substitution. Keeping the prompt definition in a separate file — rather than inline in the conversation — makes prompts easier to version, review, and reuse.

Pass the Template Prompt to the AI Assistant

This step submits the system prompt and initial user message to the API to open the conversation. The model’s response to this first message is typically a confirmation that it has received and understood the instructions rather than substantive content — it establishes the context for the rest of the conversation. See OpenAI’s prompt engineering guide for background on message roles.

Pass Text File Contents to the AI Assistant

After the conversation is initialized with the template prompt, this step passes the actual content to be worked on — typically a lesson file, draft, or document — as the next user message. The file is read as plain text and submitted directly. This is the most reliable approach for Quarto and Markdown files, which are not supported as native file uploads by the API (see Pass a File to the API below for alternatives when plain text is insufficient).

Write Output to a File for Review

After receiving a response, this step writes the model’s output to a Markdown file in a designated review folder. Saving the output as a file — rather than just printing it — makes it easier to review, diff against the original, and incorporate into the project. The filename typically includes today’s date to make outputs easy to sort and identify later.

Follow-up and Refinement

Most conversations benefit from at least one follow-up exchange after the initial response. This step is intentionally open-ended: you might ask for revisions, request a different format, probe an unclear point, or ask for a summary. Two useful patterns are creating a fork of the chat object before sending a speculative follow-up (so you can discard it without losing the main thread), and using the interactive chat interface (chat.app()) for rapid back-and-forth during refinement.

Save Chat History

Saving the full conversation history at the end of a session makes it possible to return to the conversation later, audit what was sent and received, or pass the history back to the model in a future session for continuity. The conversation templates export history in Markdown format for readability and in JSON format for programmatic use. Optionally, you can condense a long history by asking the model to summarize it before saving, which reduces the token cost of reloading it in a future session.

Render to HTML

Rendering the conversation document to HTML produces a readable, shareable record of the full session — prompts, code, and outputs together. This is useful for documentation, review, and reproducibility. Render via the Preview button in Positron or by running quarto render path/to/conversation.qmd in the terminal.

Pass a File to the API

The standard workflow above passes file contents as plain text, which works well for most cases. There are situations, however, where you need to pass the file itself — for example, when working with PDFs or other binary formats that cannot be read as plain text.

See the OpenAI documentation for full guidance:

  • https://platform.openai.com/docs/quickstart#analyze-images-and-files
  • https://platform.openai.com/docs/guides/pdf-files
Important

Only specific file formats are supported for direct upload. Common supported formats include .md, .pdf, .txt, .py, .json, .html, .yaml, and others — see the OpenAI docs for the complete list.

Quarto (.qmd) is not supported. Workarounds:

  1. Pass the file contents as plain text (the standard approach above).
  2. Convert the Quarto file to PDF before uploading.
  3. Add the file to a vector store (see Vector Stores below) — best when the file will be referenced repeatedly.

Upload and Use a File

from openai import OpenAI

client = OpenAI()

# Step 1: Upload the file
file = client.files.create(
    file=open("path/to/your-file.pdf", "rb"),
    purpose="user_data"
)

print(f"Uploaded file ID: {file.id}")
# Step 2: Use the uploaded file as input in a conversation
response = client.responses.create(
    model=model,
    input=[
        {
            "role": "user",
            "content": [
                {
                    "type": "input_file",
                    "file_id": file.id,
                },
                {
                    "type": "input_text",
                    "text": "Please summarize the main points of this document.",
                }
            ]
        }
    ]
)

print(response.output_text)
Note

Uploaded files persist in your OpenAI account until deleted. If you are uploading the same file repeatedly across sessions, consider reusing the file ID rather than re-uploading each time. File IDs can be found in the OpenAI dashboard or stored locally for reuse.

Vector Stores

A vector store lets you upload files once, index them semantically, and query them across multiple chat sessions using the file_search tool. This is the right approach when you want a document to serve as persistent, reusable context — as opposed to passing it fresh on every call.

When to Use a Vector Store

  • You have a document (or set of documents) you’ll query repeatedly
  • You want to ask questions grounded in specific file content
  • You need deterministic, source-traceable retrieval
  • The document is too large to pass as plain text in each call

Setup

Vector stores are created and managed through the OpenAI developer dashboard:

  1. Open the dashboard and navigate to Storage.
  2. Create a new vector store and give it a name.
  3. Upload your files and wait for indexing to complete (status will show as “ready”).
  4. Copy the vector store ID — you’ll use it in your code.

Query a Vector Store

from chatlas import ChatOpenAI

chat = ChatOpenAI(model=model)

vs_id = "your-vector-store-id"

# Ask a question grounded in the uploaded documents
resp = chat.chat(
    "Using ONLY the uploaded documents, answer the following question: "
    "[your question here]. "
    "If the answer is not explicitly stated in the documents, say so.",
    kwargs={
        "tools": [{"type": "file_search", "vector_store_ids": [vs_id]}]
    }
)

print(str(resp))

Request Sources

To verify which parts of the document the model used to generate its response:

resp = chat.chat(
    "[your question here]",
    kwargs={
        "tools": [{"type": "file_search", "vector_store_ids": [vs_id]}],
        "include": ["file_search_call.action.sources"],
    }
)

print(str(resp))

Multi-turn Conversations with a Vector Store

To keep file_search available across multiple turns, pass the same kwargs on each call — or set it once at the chat level:

# Option 1: pass kwargs on each turn
resp1 = chat.chat("First question", kwargs={"tools": [{"type": "file_search", "vector_store_ids": [vs_id]}]})
resp2 = chat.chat("Follow-up question", kwargs={"tools": [{"type": "file_search", "vector_store_ids": [vs_id]}]})

# Option 2: set tools at the chat level (applies to all subsequent turns)
chat.set_tools([{"type": "file_search", "vector_store_ids": [vs_id]}])
resp1 = chat.chat("First question")
resp2 = chat.chat("Follow-up question")
Note

Vector storage is not free. Currently, storage costs $0.10/GB per day, with the first GB free. See OpenAI pricing for current rates. Delete vector stores you no longer need from the dashboard.

Using Websites

There are two methods for using a website as context in a chatlas conversation. The right choice depends on whether you need the result to be reproducible.

Method 1: Add the Website to a Vector Store (Recommended)

Save the page as a PDF, upload it to a vector store, and query it with file_search. This is the most reliable approach because the content is fixed at upload time.

Steps:

  1. Open the page in a browser and save it as a PDF.
  2. Upload the PDF to your vector store (see Vector Stores above).
  3. Query using file_search.
from chatlas import ChatOpenAI

chat = ChatOpenAI(model=model)

vs_id = "your-vector-store-id"

resp = chat.chat(
    "Using ONLY the uploaded page content, answer the following question: "
    "[your question here]. "
    "Return your answer as a bulleted list.",
    kwargs={
        "tools": [{"type": "file_search", "vector_store_ids": [vs_id]}],
        "include": ["file_search_call.action.sources"],  # optional: verify sources
    }
)

print(str(resp))

Pros: Deterministic; reusable; indexed once, queried many times; source-traceable.

Cons: Requires a manual save-to-PDF step; page content is fixed at upload time and will not reflect updates to the live page.

Method 2: Live Web Search

Use the web_search tool to have the model retrieve and read a live page at query time. Add domain filtering to reduce the chance the model retrieves content from a different site.

from chatlas import ChatOpenAI

chat = ChatOpenAI(model=model)

url = "https://example.com/page-you-want-to-use"

# Basic web search
resp = chat.chat(
    f"Use web search to open and read this specific page:\n{url}\n\n"
    "Question: [your question here]\n\n"
    "Answer with a bulleted list.",
    kwargs={
        "tools": [{"type": "web_search"}],
        "tool_choice": "auto",
    },
)

print(str(resp))

To make results more deterministic with domain filtering (recommended):

resp = chat.chat(
    f"Use web search and rely on this page for the answer:\n{url}\n\n"
    "[Your question here]",
    kwargs={
        "tools": [
            {
                "type": "web_search",
                "filters": {
                    "allowed_domains": ["example.com"]
                },
            }
        ],
        "tool_choice": "auto",
    },
)

print(str(resp))

To request sources so you can verify what the model actually read:

resp = chat.chat(
    f"Use web search to answer from this page:\n{url}\n\n"
    "[Your question here]",
    kwargs={
        "tools": [{"type": "web_search"}],
        "tool_choice": "auto",
        "include": ["web_search_call.action.sources"],
    },
)

print(str(resp))
Note

If the model answers without actually searching, make the instruction more explicit: “Use web search now. Do not answer until the page has been opened.” Keep the domain filter on.

Web search is not free. Currently, it costs up to $25 per 1,000 calls. See OpenAI pricing for current rates.

Pros: Fast; no preprocessing required; reads the live, current version of the page.

Cons: Not deterministic (page content may change between calls); not reusable across sessions; not guaranteed the model actually reads the target page; not ideal for reproducible research workflows.

Reusable Online Prompts

OpenAI allows you to create and store prompt templates in the dashboard (Dashboard > Prompt), then reference them by ID in your code. The prompt text lives in OpenAI’s infrastructure rather than in your repo, and variables are injected at call time.

When this is useful:

  • You want to reduce the amount of text submitted on each call
  • You want a prompt to be shareable across team members without distributing the prompt file
  • You are iterating on a prompt and want changes to take effect immediately across all callers

Current limitations:

  • Not all models are available unless your organization is verified with OpenAI
  • The prompt text is stored externally and is not version-controlled in your repo unless you maintain a local copy

Basic Usage

Create the prompt in the OpenAI dashboard, then reference it by ID:

from openai import OpenAI

client = OpenAI()

response = client.responses.create(
    prompt={
        "id": "your-prompt-id",
        "version": "1",          # pin to a specific version
        "variables": {           # inject dynamic values
            "variable_name": "variable_value"
        }
    }
)

print(response.output_text)

Adding Follow-up Messages

To extend the conversation beyond the initial prompt, include a user message in the input field:

response = client.responses.create(
    prompt={
        "id": "your-prompt-id",
        "version": "1",
        "variables": {
            "variable_name": "variable_value"
        }
    },
    input=[{"role": "user", "content": your_content_variable}],
)

print(response.output_text)
Note

The version field pins your call to a specific version of the stored prompt. If you omit it, the call uses the latest published version, which means prompt changes in the dashboard will immediately affect your code. Pinning to a version is safer for production or reproducible workflows.

```{python}
                    "allowed_domains": ["example.com"]
                },
            }
        ],
        "tool_choice": "auto",
    },
)

print(str(resp))
```

To request sources so you can verify what the model actually read:

```{python}
resp = chat.chat(
    f"Use web search to answer from this page:\n{url}\n\n"
    "[Your question here]",
    kwargs={
        "tools": [{"type": "web_search"}],
        "tool_choice": "auto",
        "include": ["web_search_call.action.sources"],
    },
)

print(str(resp))
```

::: callout-note
If the model answers without actually searching, make the instruction more explicit: "Use web search now. Do not answer until the page has been opened." Keep the domain filter on.

# Reusable Online Prompts

OpenAI allows you to create and store prompt templates in the dashboard (Dashboard > Prompt), then reference them by ID in your code. The prompt text lives in OpenAI's infrastructure rather than in your repo, and variables are injected at call time.

**When this is useful:**

- You want to reduce the amount of text submitted on each call
- You want a prompt to be shareable across team members without distributing the prompt file
- You are iterating on a prompt and want changes to take effect immediately across all callers

**Current limitations:**

- Not all models are available unless your organization is verified with OpenAI
- The prompt text is stored externally and is not version-controlled in your repo unless you maintain a local copy

## Basic Usage

Create the prompt in the OpenAI dashboard, then reference it by ID:

```{python}
from openai import OpenAI

client = OpenAI()

response = client.responses.create(
    prompt={
        "id": "your-prompt-id",
        "version": "1",          # pin to a specific version
        "variables": {           # inject dynamic values
            "variable_name": "variable_value"
        }
    }
)

print(response.output_text)
```

## Adding Follow-up Messages

To extend the conversation beyond the initial prompt, include a user message in the `input` field:

```{python}
response = client.responses.create(
    prompt={
        "id": "your-prompt-id",
        "version": "1",
        "variables": {
            "variable_name": "variable_value"
        }
    },
    input=[{"role": "user", "content": your_content_variable}],
)

print(response.output_text)
```

::: callout-note
The `version` field pins your call to a specific version of the stored prompt. If you omit it, the call uses the latest published version, which means prompt changes in the dashboard will immediately affect your code. Pinning to a version is safer for production or reproducible workflows.
:::
