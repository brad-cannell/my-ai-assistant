API vs Browser-Based AI Workflows

Scope: The primary decision of whether a given AI-assisted task should use a browser-based interface or an API-based workflow. Does not cover choices made within the API category. Once you have decided to use the API, the next decision is which tool to reach for — see Claude Code vs. Chatlas.

Interaction modes in scope:

Mode Examples
Browser chat Claude.ai, ChatGPT web
Direct API calls chat$chat() via Chatlas in Quarto
IDE-integrated chat OpenAI Codex, Claude Code in Positron
Hybrid API/interactive chat$app() via Chatlas in Positron

Task domains: Research, grant writing, course development (Epi-Workbench), code assistance, and administrative tasks.

1. Purpose

This document provides decision criteria for choosing between browser-based AI interaction and API-based interaction for a given task. The goal is to minimize friction, maximize reproducibility where it matters, and avoid making the mode decision ad hoc.

What this SOP does not cover: Once you have decided to use the API, the next decision is which tool to use — Claude Code or Chatlas in Quarto. That choice is covered in Claude Code vs. Chatlas. The framework here applies only at the higher level: browser or API?

2. The Core Decision Rule

Before starting any AI-assisted task, ask one question:

“Will I need to find, reuse, or build on this interaction later?”

Answer Default Mode
No — exploratory, one-off, or disposable Browser
Yes — recurring, archivable, or feeds an artifact API
Unsure See Section 3

If you find yourself copying a prompt from somewhere or copying a response to somewhere, that task belongs in the API workflow.

Framing note: The goal is not to migrate all AI usage to the API. The more useful question is: “Which 20% of my AI workflows would benefit most from API-based reproducibility and control?” Start there.

3. Decision Criteria by Signal

Use these signals to resolve ambiguous cases. No single signal is decisive — weigh the pattern.

Use the Browser When:

  • The task is genuinely exploratory (“let me think out loud with Claude”)
  • You expect to use the output once and not reference it again
  • You are away from your development environment (no access to repo/terminal)
  • You need to upload a file for quick extraction or summarization and have no pipeline for it
  • The task is low-stakes and iteration speed matters more than documentation
  • You are troubleshooting or debugging interactively and expect many rapid back-and-forth exchanges

Use the API When:

  • You have used a similar prompt before, or expect to use it again
  • The output will be incorporated into a document, codebase, or artifact you are maintaining (EWB lesson, grant section, manuscript, R script)
  • Reproducibility matters — you may need to document what was sent, what parameters were used, and what was returned
  • The task involves file context that is structured and reusable (formatting guidelines, CST examples, style rules)
  • You are building or maintaining a pipeline (EWB lesson creation, batch content review)
  • You want parameter control (model selection, temperature, system prompt)
  • The task involves sensitive research content where better data handling terms apply
  • You are teaching or demonstrating AI usage and want a documentable, transferable example
  • The task would benefit from a model other than the browser default — e.g., a larger context window for full document analysis, or a faster/cheaper model for batch or repetitive work
  • The task is high-volume or cost-sensitive and strategic model selection would meaningfully reduce cost or processing time

Boundary Cases

Situation Recommendation
Quick code fix, one function Browser — unless it’s part of a documented pipeline
Drafting an email from a template API — prompt is reusable; output may be archived
Uploading a PDF for a one-time summary Browser
Uploading a file as structured context for ongoing work API
Brainstorming grant aims Browser — then migrate key outputs to API doc or notes
Refining specific aims language API
Asking “does this paragraph make sense?” Browser
Full manuscript section review API

4. Security Considerations

These practices apply regardless of which mode you use, but several are especially relevant to the mode decision.

  • Never paste into any AI interface: PHI, SSNs, IRB-restricted data, financial data, or any content governed by a Data Processing Agreement or Business Associate Agreement. Neither standard browser nor standard API access is an appropriate channel for this content without specific institutional agreements in place.
  • Check institutional policy: TCU and funding agencies may have guidance or requirements governing which AI tools are permissible for specific research or course contexts. Verify before using either mode for grant-related or IRB-adjacent work.
  • API keys: Store in environment variables (.Renviron for R, .env for Python). Never commit keys to Git. Rotate periodically and immediately if exposure is suspected.
  • Browser security: Audit extensions regularly. Extensions with broad permissions can access conversation content. Consider a dedicated browser profile for AI work involving sensitive material.
  • Opt out of training data use: Disable conversation use for model training in browser settings where available. API usage under standard Anthropic terms does not use your data for training by default.
  • Local logging: If storing API responses locally (in your ai/ folder or elsewhere), be aware that those files contain the full prompt and response. Apply appropriate access controls if the content is sensitive.

5. Maintenance

Review and update this SOP when any of the following occur:

Trigger Action
A task type is repeatedly misclassified Add or revise a row in the Boundary Cases table (Section 3)
A new recurring task type emerges Assess which mode it belongs to; add to the appropriate signal list
The core decision rule proves insufficient for a new class of tasks Revise Section 2 and document the rationale
A downstream SOP (API workflow, browser workflow) changes in ways that affect mode choice Update decision criteria here to stay consistent
This SOP has not been reviewed in 6 months Re-read and confirm it still reflects actual practice
