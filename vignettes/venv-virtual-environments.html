Overview

This vignette covers creating and managing Python virtual environments using the built-in venv tool in Positron. For a modern, cross-machine alternative with lockfile support, see vignettes/uv-virtual-environments.qmd.

Why Virtual Environments Matter

When Positron suggests creating a virtual environment, it’s saying:

“Let this project have its own Python and package setup, so changes here don’t accidentally break other projects — or vice versa.”

A venv is:

  • A copy of the Python interpreter
  • Plus a private site-packages/ folder
  • Scoped to one project

Nothing magical — just isolation.

Why This Matters for a Chat / API Project

If you install packages globally:

pip install openai pydantic rich

Those versions become shared by every Python script on your machine — easy to accidentally upgrade or downgrade, and hard to reproduce later. This becomes painful when one project needs pydantic<2 and another needs pydantic>=2. A venv prevents this class of problems entirely.

Create a Virtual Environment

Method 1: Command Palette (Recommended)

This is the fastest way to have Positron create and automatically register the environment for your current workspace.

  1. Open your project folder in Positron.
  2. Open the Command Palette: Cmd+Shift+P (macOS) or Ctrl+Shift+P (Windows/Linux).
  3. Type and select Python: Create Environment….
  4. Select venv as the environment provider.
  5. Choose the Python interpreter version you want to use.

Positron will create a .venv/ folder in your project root and automatically set it as the active interpreter for your workspace.

Method 2: Manual Terminal Setup

If you prefer the command line or need specific configurations:

  1. Open the integrated terminal in Positron.
  2. Create the venv:
    • macOS/Linux: python3 -m venv .venv
    • Windows: python -m venv .venv
  3. Activate the venv:
    • macOS/Linux: source .venv/bin/activate
    • Windows: .venv\Scripts\activate
  4. Connect to Positron: click the Python interpreter version in the top-right corner (or use Python: Select Interpreter in the Command Palette) and select the path to the new .venv.

Activate the Environment

Before installing or using packages in your venv, activate it in the terminal:

source .venv/bin/activate   # macOS/Linux
.venv\Scripts\activate      # Windows

Once activated:

  • python → uses the venv Python
  • pip → installs into the venv
  • sys.path → resolves packages automatically

Your code does not need to reference .venv at all.

Install Packages

With the venv activated, use pip install normally:

pip install keyring
pip install openai
pip install jupyter
pip install -U chatlas

You may need to restart the kernel after installing packages. To restart the Python kernel in Positron, open the Console pane, select the kernel you want to restart on the right, and click the circular arrow button.

File Paths

If you’re seeing paths like /Users/.../project/.venv/lib/python3.12/site-packages, that’s just Python being transparent — not something you should ever type manually.

Do this once per project and forget about it:

cd your-project
python3 -m venv .venv
source .venv/bin/activate

Then select the .venv Python interpreter in Positron. You’ll only notice it again when things don’t break.

.gitignore Recommendations

Exclude the virtual environment from version control and sync tools:

.venv/
__pycache__/
Note

Virtual environments are not portable across machines — absolute paths are embedded in the environment at creation time. If you work across multiple computers with a synced folder (e.g., Box, Dropbox), do not sync .venv/. See vignettes/uv-virtual-environments.qmd for a workflow designed specifically for this scenario.
